The Evolution DNS

This standard has existed for a long time in fact the first RFC about DNS was published in 1987. But a lot has changed from the first publication. The original idea was great, it has work and it continues to work for more then 30 years with the ridiculous internet growth but it has some limitations which were solved with more recent standards.

Before DNS

Before dns was release and adopted host name to address mappings were maintained by the Network Information Center (NIC) in a single file (HOSTS.TXT) which was FTPed by all hosts.

First DNS Standard

In the first version of dns when sending a query the data is transmitted over UDP this is done mainly for performance reasons. UDP is fast and with small size packets like DNS the packet can fit in single MTU in most cases.

Later DNS over TCP was added and a lot of server provided it as an optional dependency. This provides foundation support for DNSSEC for DoT.

Still even the standard was made long time ago in most case we still use UDP due to lack of software support or lack of understanding.

You may ask why should we change this when it works? If you ask this you are right and wrong. Yes in some cases is okay to use this standard if you have full control of the network but if you don’t the security and privacy aspect are more important then the hardly noticeable performance gains.

DNSSEC

As you may have guest DNSSEC was added to enhance the security, in RFC-4033 says:

The DNS security extensions provide origin authentication and integrity protection for DNS data, as well as a means of public key distribution. These extensions do not provide confidentiality.

That means DNSSEC provides validation that nobody is messing with the dns query by adding cryptographic signatures to DNS records. This solves almost all dns attacks and you can be sure that nobody messed with the response that you get from the server. When using DNSSEC use of TCP is recommended since this increases the query size and UDP may not be so reliable.

But still this does not provide encryption so anyone that can read the request can parse the query. This does not help at all with the privacy problem.

DoT and DoH

Does two standards solve all of the security and privacy problems from before.

DoT (DNS over TLS) is just a normal dns over tcp but all the data is encrypted with TLS. It usually uses 453 port.

DoH (DNS over HTTPS) is dns query passed through HTTPS connection. It uses the standard 443 https port.

Both are end-to-end encrypted no need for additional checks or DNSSEC.

This is great we have a new standards that has great security, privacy and with proper caching there will be no performance difference. The problem is as every other networking standard, the adoption is slow. Most of the mobile devices that run Android or IOS have option for DoT or DoH as well as most modern browsers but it’s almost never the default option.

Unless you have a router that support encrypted dns and can act as a DNS server (which is the easiest option) I would suggest checking all you devices settings and setup DoT or DoH.

You may have notices why do we have DoH when DoT has the same benefits and is more efficient (it dose not need to send the http headers). Well the reason is that DoH uses the same port as any HTTP app and sys admins can block the port 53 and 453 but cannot block 443 without making the internet unusable. Also it provide better privacy when you make a dns request over DoT for example the ISP can’t see what DNS request you are making but it can see that you are making one and you are encrypting it but if you use DoH the ISP will see a regular HTTPS request and it wont know that it is a DNS request.
